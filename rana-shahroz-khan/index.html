💡 What I find most interesting

One agenda runs through his papers: make large models cheap to adapt — and stress-test where adaptation and coordination break. Three works are about cheap / portable / generated adaptation (PortLLM, ORAL, and CAR-LoRA all live in the LoRA + “evolving LLMs” world); the others push the same efficiency instinct into trustworthiness (Agents Under Siege) and reward-free training (TMS).

The instinct has clear roots: his undergrad work under Soheil Kolouri was optimal transport + model compression (PRANC, Linear Optimal Partial Transport) — the “compact representations in parameter space” idea that PortLLM and ORAL inherit. So the multi-agent-attack and SFT papers aren’t detours: they’re the trustworthiness and training-efficiency arms of one efficiency-of-deployed-LLMs program. PortLLM: Training-Free, Portable Model Patches 1st

Carry a fine-tuned model’s knowledge forward to a newer base-model version by saving the fine-tune as a small reusable patch (a LoRA delta) and just adding it — no retraining.
🔑 Key finding. You can skip re-fine-tuning entirely — just add the OLD task patch to the NEW base model. It matches real re-fine-tuning (and on GSM8K beats it, 41.32 vs 34.95) at 0 trainable parameters, ~12× less GPU memory, and ~4897× fewer GPU-hours.

Intro — why

LoRA fine-tuning freezes W₀ and learns a low-rank ΔW = B·A (cheap, but still needs GPU training). When the provider ships an updated base model (modeled as continued pretraining, θ' = θ + Δθ), you’d normally re-fine-tune — costly, recurring, and sometimes impossible if the original data is access-locked. PortLLM asks: can the patch from the old model just be reused on the new one?

Method — how

Setup. You already fine-tuned the old base θ for your task with LoRA, giving a patch Δθi = B·A. Now the provider ships a better base θ′ (modeled as continued pretraining, θ′ = θ + Δθ). Baseline — the gold standard is to actually re-fine-tune the new model to get its own patch, which needs GPUs and the original data:

θ′i = θ′ + Δθ′i (re-train on the new base)
What PortLLM optimizes away. It never computes Δθ′i at all — it reuses the old patch with a plain weight-add:
θ′i ≈ θ′ + Δθi (no training — runs in seconds)
Why this is allowed. The exact answer is θ′i = (θ′ + Δθi) + R, where the residual you drop is R = Δθ′i − Δθi. R is negligible because both task patches are low-rank (LoRA) while the base models are full-rank — empirically R’s Frobenius norm is ~136× smaller than the naive-add term, so ignoring it barely changes the output.

Results

Mistral-7B / Llama2-7B / Llama3.1-8B / Gemma2-9B, 7 tasks, zero-shot. Lands within ~1 point of real re-fine-tuning on most tasks and beats it on WNLI and GSM8K (41.32 vs 34.95; the un-patched updated model scored only 15.16). Gap ≤ 0.73%, sometimes +5.53%. Cost vs LoRA fine-tuning: 0 trainable params, ~12.2× less GPU memory, ~4897× fewer GPU hours (merge runs in seconds).

Discussion & future

Future: extend across different architectures (via model merging). It already works even if the provider’s continued pretraining was full-weight, not LoRA.

⚠ Most important thing missing: Only validated when the same model lineage evolves via continued pretraining (same architecture & weight shape). A real version jump that changes architecture/tokenizer/hidden-size has no coordinate correspondence — you can’t add the old delta. So PortLLM = cheap portability across successive same-architecture checkpoints, not a universal patch surviving an architectural change (the paper’s own future work names this as open).

Rana M. Shahroz Khan (1st), P. Li, S. Yun, Z. Wang, S. Nirjon, C.-W. Wong, T. Chen. ICLR 2025. arXiv · evidence

ORAL: Generating Large-Scale LoRAs via Conditional Recurrent Diffusion 1st

A diffusion model that GENERATES task-specific LoRA adapter weights from a text prompt + a target-model description, instead of training each one — scaling to 7B-parameter LLMs.
🔑 Key finding. A diffusion model can generate a working LoRA adapter from a text prompt + a description of the target model — matching trained LoRAs (e.g. GSM8K 34.67 vs 32.04) and, unlike prior weight-generators, scaling all the way to 7B-parameter LLMs.

Intro — why

Instead of producing a LoRA adapter by gradient descent, synthesize its weights with a generative model. Motivation = evolving LLMs again: when the base model updates, regenerate adapters from a prompt rather than re-training every task adapter. Prior weight-generation methods couldn’t get both scalability and controllability.

Method — how

Setup. A LoRA adapter is just a low-rank update ΔW = B·A, normally obtained by training (gradient descent on task data). Baseline:

(B, A) = argmin Ltask( W0 + B·A ) (train each adapter from data)
What ORAL changes. It replaces training with generation — a diffusion model produces the adapter weights, conditioned on what you want:
(B, A) = Diffusion( noise | c ), c = [ cmodel ; ctext ]
where ctext is the task description and cmodel encodes the target model. The scaling trick (why prior methods couldn’t reach 7B): slice the huge weight matrix into a sequence of fixed-size tokens, let a Mamba recurrence carry state along it, and have diffusion denoise each token — so it never has to flatten one giant vector. Why condition on cmodel: it is exactly what lets one generator emit adapters for different / evolving base models straight from a prompt.

Results

14 tasks (7 language, 4 vision, 3 multimodal), 5 base models incl. Mistral-7B, Qwen-VL-7B, SD 2.1; scales to 7B (prior baselines did not). Generated LoRAs are ≈ or > trained ones (e.g. GSM8K 34.67 vs 32.04; image FID mostly wins). Cross-model transfer: train on base versions t=0,1,2 → generate adapters for unseen evolved versions t=3,4.

Discussion & future

No dedicated limitations section. Future: interpretability of generated parameters.

⚠ Most important thing missing: It doesn’t beat a single trained LoRA (≈ tied). The value is amortization + transfer: you must first collect many real trained LoRAs to train the generator (training is front-loaded once), then produce new adapters by prompting; the real win is regenerating adapters when the base model evolves. If you only need one adapter for one fixed model, plain LoRA training is simpler.

Rana M. Shahroz Khan (1st), D. Tang, P. Li, K. Wang, T. Chen. EMNLP 2025 Findings. arXiv · evidence

Agents Under Siege: Optimized Prompt Attacks on Multi-Agent LLM Systems 1st

A jailbreak that splits a malicious request into pieces, routes them through a team of cooperating agents along the safest path, and reassembles the attack at the target — defeating filters that only see harmless fragments.
🔑 Key finding. Splitting one jailbreak across cooperating agents and routing the pieces along the least-guarded path reaches 72.6% attack success (vs 1.7% for standard GCG) and cuts the best safety filter’s F1 by ~30% — because no single guarded link ever sees the whole payload.

Intro — why

A multi-agent LLM system wires several LLMs that pass messages over a graph. Almost all safety work studies a single agent — but multi-agent systems have new attack surfaces: inter-agent messages, per-link token-bandwidth limits (so chop a request below filter size), latency/asynchrony (messages arrive in any order), and partial defenses (some paths are barely guarded). These “pragmatic” constraints become attacker advantages.

Method — how

Setup. A multi-agent system is a graph of LLMs passing messages; each link has a token-bandwidth limit, messages can arrive out of order, and only some links are guarded by a safety filter. Threat model: partial knowledge of the graph + gradient access to the target. Two ingredients exploit the structure. (1) Routing (MFMC). Treat the network as a flow graph where each link has capacity F (bandwidth) and cost G (detection risk), and solve for the cheapest-to-detect route and chunk sizes:

minimize Σ G(u,v)·f(u,v) subject to 0 ≤ f(u,v) ≤ F(u,v)
i.e. send the payload through the least-guarded, highest-bandwidth path, chopped small enough to slip under each link’s filter. (2) Reorder-proof chunks (PIEL). Since chunks may arrive in any order, optimize them (via GCG) so the attack fires under every permutation by averaging the evasion loss over orderings:
L(C) = (1/K!) Σπ −log p( harmful output | reordered chunks )
(stochastic S-PIEL samples a few permutations). MFMC decides how to chop and where to route; PIEL makes each fragment reassemble into a jailbreak regardless of order.

Results

Up to 7× over the best baseline. Llama2-7B / JailbreakBench: ASR 72.6% vs GCG 1.7%. Most vulnerable Mistral-7B (81.2%), most resistant DeepSeek-R1-Distilled (41.3%); range ~38–84%. Defeats Llama-Guard / Prompt-Guard (best guard’s F1 drops ~30%). Complete-graph topologies most vulnerable (~78%), chains most resilient (~60%).

Discussion & future

Open-source models ≤9B only; assumes partial topology/defense knowledge; static defenses; text-only. Proposes no concrete defense — argues single-agent filters are the wrong abstraction; calls for multi-agent-specific defenses. Red-team ethics disclosure.

⚠ Most important thing missing: Every result is on small open-source models under favorable assumptions (attacker knows the topology + where defenses sit), so it does not show production systems (GPT-4/Claude, hidden topology) are breakable at these rates. Read 72–84% as an upper-bound demo. The transferable contribution is conceptual: structural properties (bandwidth, asynchrony, partial defense) are exploitable attack vectors, and per-message single-agent filters are the wrong abstraction because they never see the whole payload.

Rana M. Shahroz Khan (1st), Z. Tan, S. Yun, C. Fleming, T. Chen. ACL 2025 (Main). arXiv · evidence

TMS: Trajectory-Mixed Supervision for Reward-Free, On-Policy SFT 1st

Fine-tune an LLM to get most of RL’s stability benefit WITHOUT a reward model — by training it on samples drawn from its own past training checkpoints.
🔑 Key finding. Train on the model’s own answers from earlier checkpoints instead of fixed labels: cross-task forgetting drops from −26 to −41 points (plain SFT) to just −2 to −3 — close to RL — with no reward model.

Intro — why

Plain SFT (fixed labels) is cheap but brittle — it forgets. RLHF (reward model + RL) retains capability better but is costly and unstable. The root cause TMS names is Supervision Mismatch: as the model trains, its policy drifts away from the static labels, causing mode collapse and forgetting. Avoiding a reward model is desirable (less cost, no reward-hacking, no preference data).

Method — how

Setup. Plain SFT trains on fixed (question → label) pairs. As the model trains, its policy drifts away from those static labels (the supervision mismatch) → mode collapse and forgetting. RL fixes this but needs a reward model. Baseline (SFT):

minimize −log πθ( y* | x ) (imitate a fixed label y* the model drifts from)
What TMS changes. During one SFT run, save T checkpoints and have each checkpoint generate its own answer ŷ(t)(x). Train on a mixture of those self-generated answers (optionally blending the labels q with weight α):
qα = α·q + (1−α)·m, m(·|x) = (1/T) Σt ŷ(t)(x) (α = 0.25, T = 10)
Why it works. The targets are text the model itself plausibly produces, so supervision stays near its own support — the stabilizing property of on-policy RL — yet the signal is just “imitate your own past samples” (next-token loss), so there is no reward model, verifier, or RL loop. Early checkpoints add diversity (anti-collapse), late ones add refinement — a built-in curriculum.

Results

Qwen-2.5 (1.5/3/7B), LLaMA-3.1-8B; math / instruction tasks. Matches SFT target accuracy while nearly eliminating forgetting: average cross-task drop is −26 to −41 for SFT vs only −2.3 to −2.9 for TMS (GRPO/RL: −0.9 to −1.9). Safety retention also improves toward RL. Cost: harvest 10 checkpoints + one student pass — no reward models or verifiers.

Discussion & future

Largest gains on non-unique solution spaces (math, instruction-following); modest on rigid formats (MMLU). Optimal T ≈ 8–10. Safety gains require careful data curation.

⚠ Most important thing missing: TMS does not beat RL — it only narrows the gap; “approaches RL retention” can be mis-read as “matches RL.” The paper is explicit that RL is the gold standard and TMS incurs a modestly larger cross-task drop than GRPO. Correct takeaway: a cheap, reward-free drop-in that recovers most of RL’s stability at near-SFT cost. Secondary risk: targets are the model’s own generations, so latent base-model biases can be reinforced.

Rana M. Shahroz Khan (1st), Z. Liu, Z. Tan, C. Fleming, T. Chen. 2026 preprint venue unconfirmed. arXiv · evidence

Your turn: discussion questions

  1. Generate vs train adapters. ORAL generates LoRAs and PortLLM reuses them; both only ≈ match a trained LoRA. When is “cheaper, equally good” worth a whole new method — and when is it premature optimization?
  2. Is per-message filtering doomed? Agents Under Siege shows fragment-and-route attacks defeat single-agent guards. What would a defense that “sees the whole payload” across a multi-agent system actually look like — and what does it cost?
  3. Cheap stability vs the real thing. TMS recovers ~most of RL’s retention with no reward model, but training on your own outputs can amplify your own biases. When is “imitate your past self” a safe training signal, and when isn’t it?
