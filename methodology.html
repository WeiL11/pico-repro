Methodology — the living research method

This page tracks the method itself — how our self-driving research-and-teach workflow works now, and how it has improved. Each research direction applies the current method.

Why this page exists

The point of this project is not any single paper — it is to build an efficient agent workflow that can do research largely on its own and turn it into cited learning notes for undergraduates. So the method is the real artifact, and this page is where we keep it and watch it improve.

Each research direction (e.g. PICO, Alisa Liu) has its own methodology.html describing how that specific study was built. This top-level page is the general method + its changelog.

The current method (v0.6) — two passes under four principles

  1. Explain the concept well — each paper gets Intro (why) / Method (how) / Results / Discussion, jargon defined.
  2. Align every statement with the research, with a source on EVERY argument — end to end. Read primary sources; never assert from memory; carry a citation on each claim from the agent outputs through to the final HTML, then verify each annotation once. Flag the unverified.
  3. Every agent answers "what is the most important thing missing?" — which becomes a box per paper.
  4. Open each content page with a personal "what I find most interesting" take — the recurring problem the researcher tackles + what they seem drawn to.

v0.6 additions (adopted from the retrospective)

  • Pipeline, not barrier — start Pass 2 as soon as the paper list exists; don't wait for all of Pass 1.
  • Per-agent budget — cap each agent's fetches/time; prefer abstract + HTML + key tables over full PDFs.
  • Slimmer Pass 1 — 1–2 agents instead of 3 (de-duplicate the broad map; fold summaries into Pass 2).
  • Per-claim provenance, verified once — every argument carries a source from the agent outputs to the final HTML; a final pass checks each annotation.
  • Evidence in the repo — each agent's full deep-read saved as _research/<paper>.md so every claim is traceable.
  • Decision log — record why these papers / choices, in the repo (not just chat).
  • Completeness critic — an agent asks "what's missing / why these papers / what counterexamples?"
  • Output-quality gate — an LLM-judge / QA check that the page actually teaches (clarity + accuracy).
  • Cross-run diff — when two runs of the same subject exist, diff them as a free QA signal.
  • Telemetry — record time / token cost per step and per direction.
  • Batched status — summarise once at the end, not on every agent completion.
  • Citation counts — a sanctioned source (e.g. Semantic Scholar) or explicit omission with a note.

Not adopted: skipping re-identification for already-known subjects; a separate whole-page consistency re-read (covered by per-claim verification).

How the method has progressed (changelog)

VersionWhat changedWhat it taught us
v0.1
PICO build		 One research pass (web fetches), then build demos in a run → inspect → fix loop, with verification done at the end. Verifying late is costly: two demos first showed the opposite lesson and one experiment contradicted the paper, forcing rework. → verify earlier; don't ship a misleading result.
v0.2
PICO review		 Added an adversarial fact-check sub-agent that checked ~26 claims against the paper. It caught 4 wrong/over-stated claims (e.g. the Haar "19.5%" is vs pixel-reshuffle, not stride-2 conv). → an independent verify pass works.
v0.3
Alisa pass 1		 Parallelised research into a 3-agent fan-out (list / summaries / arc) and moved verification before writing (a verify gate). Caught a misattribution (Infini-gram is not hers) before it shipped. → front-load verification; fan out for speed.
v0.4
Alisa pass 2		 Added a per-paper deep-read fan-out (one agent per paper → Intro/Method/Results/Future + "what's missing"), formalised the three principles, foldable <details> output, and the "what I find most interesting" opening. Depth + scannability together; every claim cited; the most-missing question made explicit.
v0.5
fresh-run reproduction · current		 Ran the full v0.4 pipeline completely from scratch on a researcher (from only a Scholar URL, no prior notes) as a reproducibility test — 3-agent Pass 1 → verify gate → 6-agent Pass 2 → folded HTML, applied from the very start (no shallow first draft). The method reproduces: independent agents re-identified the researcher, disambiguated same-name people, and tightened details (e.g. confirmed co-first authorships from the PDFs). Direction: alisa-liu-v2.
v0.6
retrospective · adopted, to apply next		 From a process review: pipeline (no barrier), per-agent budgets, slimmer Pass 1, per-claim provenance verified once, evidence + decision log in the repo (_research/), a completeness critic, an output-quality gate, cross-run diff, telemetry, and batched status. Faster (pipeline + budgets), more traceable (every claim sourced & verified), and self-checking (critic + quality gate + diff). Applies from the next direction.
Next planned improvement: apply v0.4 from the start of the next direction (no shallow first draft), and consider a final "completeness critic" agent that asks what the whole note still omits.
