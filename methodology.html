How these PICO notes were made

A behind-the-scenes report on the research & build process — the workflow, the tools, the bugs I caught, and how I verified the claims before teaching them.

TL;DR

I treated this like a small research project, not a one-shot write-up. The work ran in six phases — Research → Scope → Build → Write → Verify → Publish — with a tight build–run–inspect–fix loop inside “Build,” and an adversarial fact-check inside “Verify.” The guiding rule throughout: run it and verify it before you teach it.

This matters because several things that “sounded right” turned out to be wrong when actually executed: two demos initially showed the opposite of the lesson, one experiment contradicted the paper at small scale, and a fact-check caught four numbers/claims that needed correcting. None of those would have surfaced without running and checking.

The overall workflow

The whole project followed this loop. The dashed arrow is the important part: findings from Verify (and bugs from Build) fed back and changed earlier work — it was not linear.

1 Research — read the paper from primary sources

I did not answer from memory (the paper is recent — arXiv:2605.05148, 2026). I pulled facts from primary sources with web tools:

  • WebSearch → identified the paper as Apple’s PICO and found the arXiv ID, Apple ML Research page, and project page.
  • WebFetch on the arXiv HTML (not just the abstract) → extracted the concrete details: mean-scale hyperprior backbone, Haar resampling, ConvScale, learned quantization width, the one-shot context model, the loss cocktail, and the ablation table.
  • Re-fetched specific numbers on demand later (model sizes, NAS channel counts) when a question needed them.

Output of this phase: a structured mental model of what PICO is and which numbers the paper actually reports — the raw material for everything downstream.

2 Scope — plan, ask, and re-scope when redirected

A full reproduction is impossible free/local (proprietary NAS, ~120k-image training, on-device runtime). So I scoped deliberately rather than guessing:

  • Entered plan mode, checked the environment (Python 3.13, torch 2.9.1 with Apple-MPS, matplotlib), and wrote a plan file.
  • Used clarifying questions to fix the scope (focus / compute budget / which losses).
  • Re-scoped twice when you redirected: first to “understand the concept + simulate,” then to “study, teach via an HTML page, with Python demos.” The plan file was rewritten each time.
Key judgement call: aim for faithful-in-spirit at small scale — isolate each idea the paper says matters and simulate that, instead of pretending to reproduce SOTA numbers.

3 Build — the run–inspect–fix loop (where the real work happened)

Each demo went through this cycle. The crucial habit: after writing code I ran it and looked at the actual figure/output (I literally read the generated PNGs), then asked “does this show the lesson, or the opposite?” If it failed, I diagnosed the root cause before patching.

Lesson: a plausible explanation is not evidence. Each of these would have shipped as a confident — and wrong — teaching point if I had not run it and looked at the result.

4 Write — grow the HTML in layers

The teaching page was not written once; it accreted as you asked for more depth. Roughly in order:

  1. Core concept sections (problem, pipeline, Haar, quantization, entropy, what-matters, perceptual) each pairing a paragraph with its demo figure.
  2. A “how PICO differs from earlier codecs” section.
  3. A history timeline (JPEG → learned → perceptual → PICO), then a concrete JPEG walkthrough with real figures.
  4. A step-by-step “how PICO works” encode/decode walkthrough + a which-step-is-ML table + a model-size table.
  5. An evaluation / scoring explanation (training objective vs. how victory is judged).
  6. Framing for the audience (a study-group banner) and trimmed discussion questions.
  7. Citations threaded through the text with a numbered References list.

Throughout, every edit re-opened the page and I validated structure programmatically (balanced section tags, every [n] citation resolving to a real reference).

5 Verify — adversarial fact-check against the paper

Self-checking is not enough for claims I’m about to hand to students, so I spawned independent sub-agents to research and to try to falsify the notes.

What the fact-check changed

The agent verified 26 specific claims against the arXiv paper. 22 were correct; 4 needed fixing, and I applied each to the notes and the figure:

  • Haar “19.5%” is vs a naive pixel-reshuffle — vs a stride-2 conv it is only 8.9%. (Relabelled everywhere + in the chart.)
  • Tiles are non-overlapping 504×504, padded to 512 with a 4-px contextual margin — not “a small overlap.”
  • “71 quality levels” are 8 coarse levels interpolated to 71, not 71 native.
  • 230/150 ms is best-case (“as fast as”), not a fixed latency.
This phase is the difference between “a confident page” and “a page you can hand to students.” The most load-bearing number on the site (the headline 19.5% lever) was the one most in need of nuance.

6 Publish — version control & GitHub

  • Initialised a git repo, added a .gitignore (excluded the downloaded Kodak cache, which isn’t mine to redistribute, and Python caches).
  • Committed source + the generated figures (so the page renders without running anything) and pushed to github.com/WeiL11/pico-repro on main.

Principles that kept it honest

  • Run it before you teach it. Every quantitative claim in a demo is something I executed and saw.
  • Don’t ship a misleading result. When a tiny-scale experiment contradicted the paper, I removed it rather than dress it up.
  • Cite primary sources, then verify them. Web-fetch the paper; don’t trust memory for a 2026 result.
  • Adversarial check. Use independent sub-agents to try to falsify the claims, not just confirm them.
  • Free / local by default. Everything runs on a laptop CPU/MPS; no paid services.
  • Be explicit about scope. A prominent box states what is reproduced vs. only explained.
